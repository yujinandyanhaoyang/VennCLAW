# OpenClaw + Claude Code 超强教程：一个人就能搭建完整的开发团队！
发布时间：2026年2月25日
发布方：Datawhale

## 一、系统核心效果：一人抵一开发团队
这是2026年1月真实发生的B2B SaaS产品开发实践，独立开发者通过OpenClaw+Codex/Claude Code搭建的AI Agent系统，实现了开发效率的极致提升，核心数据如下：
- 单日最高94次代码提交，平均每日50次提交
- 30分钟内完成7个PR（代码合并请求）
- 客户需求**当天交付上线**，直接转化付费用户
- 成本可控：重度使用每月仅$190（Claude $100 + Codex $90），新手起步$20即可运行
- 开发者可脱离编辑器，一天开3个客户会议仍能完成海量开发工作

核心转变：从**直接管理Claude Code/Codex**，变为**通过OpenClaw管理AI管家，再由管家调度各类代码模型**，实现小到中等复杂度任务的全自动化。

## 二、为什么需要OpenClaw做编排层？
Codex和Claude Code单独使用存在**根本性上下文限制**：模型的上下文窗口固定，只能二选一填充内容，导致开发中出现关键问题：
- 塞满代码库 → 无空间存放业务上下文（客户需求、产品定位等）
- 塞满客户历史 → 无空间存放代码相关信息（类型定义、API文档等）

最终单独使用时，模型会出现**业务认知缺失**：不知道功能对应的客户、类似需求的失败原因、产品设计原则，仅能根据当前代码和简单prompt工作。

OpenClaw的核心价值：作为**编排层**架设于开发者和所有AI工具之间，解决上下文分工问题，类比关系为：
> Codex/Claude Code = 专业厨师（只管做菜）
> OpenClaw = 主厨（掌握客人口味/库存/定位，下达精准指令）

## 三、双层系统架构：编排层 + 执行层
系统分为两层，各司其职且安全边界清晰，执行层永远无法接触客户敏感信息，仅能获取完成任务的**最小上下文**。

### 3.1 两层核心职责对比
| 层级 | 核心角色 | 持有的上下文 | 负责的核心工作 |
|------|----------|--------------|----------------|
| 编排层 | OpenClaw（作者命名为Zoe） | 业务上下文：客户数据、会议记录、历史决策、成功/失败模式 | 理解需求、拆解任务、生成精准prompt、选择适配AI、监控进度、处理失败重试 |
| 执行层 | Codex / Claude Code / Gemini | 代码上下文：代码库、类型定义、测试文件、API文档 | 写代码、运行测试/构建、提交PR、响应代码审查反馈 |

### 3.2 编排层（OpenClaw）核心能力
1. 自动同步并读取Obsidian笔记中的所有会议记录
2. 以**只读权限**访问生产数据库，获取客户配置信息
3. 拥有管理员API权限，可直接给客户充值解除使用阻塞
4. 根据任务类型智能选择适配的代码Agent（Codex/Claude Code/Gemini）
5. 监控所有Agent运行进度，失败后分析原因并**动态调整prompt重试**
6. 任务完成后通过Telegram向开发者发送通知

### 3.3 执行层（各类Code Agent）核心能力
1. 读写代码库，完成代码编写/重构
2. 独立运行代码测试、项目构建
3. 自动提交代码、创建PR
4. 响应代码审查（Code Review）的反馈并优化代码

### 3.4 核心能力对比
#### Zoe（OpenClaw）
**擅长**：理解业务原因、判断客户优先级、调研分析、生成精准prompt
**不擅长**：编写实际代码、设计文件结构、遵循代码规范
**持有信息**：客户CRM、会议记录、竞品情报、付费信息、记忆系统（MEMORY.md/日常笔记/历史决策）

#### Codex（核心代码Agent）
**擅长**：理解代码库、遵循编码规范、编写可运行代码、执行测试
**不擅长**：理解业务价值、功能优先级判断、长期记忆
**持有信息**：代码库约定、风格指南、设计文档、功能规格、API模式、代码目录

## 四、完整工作流：从客户需求到PR合并的8个步骤
以**企业客户需求：复用已配置设置，实现团队内共享**为例，讲解全自动化开发流程，整体耗时1-2小时，开发者实际投入仅10分钟。

### 背景
企业客户致电希望实现配置的团队共享，所有通话记录自动同步至Obsidian，OpenClaw已提前获取完整业务上下文。

### 步骤1：客户需求 → OpenClaw理解并拆解
1. OpenClaw（Zoe）与开发者协作，将需求拆解为**开发模板系统，支持保存/编辑现有配置**
2. Zoe自动完成三件事：
   - 调用管理员API给客户充值，解除使用限制
   - 从生产数据库拉取客户现有配置
   - 打包所有业务上下文，生成精准prompt并启动Codex

### 步骤2：启动代理，创建隔离开发环境
Zoe为任务创建**隔离的开发环境**，避免影响主分支，核心操作：
1. 创建独立git worktree（分支环境）
2. 启动tmux后台会话，让Agent在后台运行
3. 生成JSON任务记录文件，记录任务ID、Agent类型、分支、状态等信息

#### 核心命令示例
```bash
# 创建git worktree并切换分支
git worktree add ../feat-custom-templates -b feat/custom-templates origin/main
cd ../feat-custom-templates && pnpm install
# 启动tmux后台会话运行Codex Agent
tmux new-session -d -s "codex-templates" \
  -c "/Users/elvis/Documents/GitHub/medialyst-worktrees/feat-custom-templates" \
  "$HOME/.codex-agent/run-agent.sh templates gpt-5.3-codex high"
```

#### tmux核心价值：中途干预
无需杀掉Agent重来，可直接向后台会话发送指令调整开发方向：
```bash
# 纠正Agent开发方向
tmux send-keys -t codex-templates "停一下。先做 API 层,别管 UI。" Enter
# 补充代码上下文
tmux send-keys -t codex-templates "类型定义在 src/types/template.ts,用那个。" Enter
```

#### 任务记录JSON示例
```json
{
  "id": "feat-custom-templates",
  "tmuxSession": "codex-templates",
  "agent": "codex",
  "description": "企业客户的自定义邮件模板功能",
  "repo": "medialyst",
  "worktree": "feat-custom-templates",
  "branch": "feat/custom-templates",
  "startedAt": 1740268800000,
  "status": "running",
  "notifyOnComplete": true
}
```

### 步骤3：自动监控，低成本检查任务状态
1. 每10分钟通过cron任务检查所有Agent状态，**不直接询问Agent**（节省token），仅检查客观事实：
   - tmux会话是否存活
   - 是否已创建PR
   - CI/CD执行状态
   - 是否需要重启（最多重试3次）
2. 监控脚本100%确定性，仅在需要人工介入时通知开发者，基于**改进版Ralph Loop**实现。

### 步骤4：Agent创建PR，定义“完成”标准
1. Agent完成代码编写后，自动提交、推送代码，并通过`gh pr create --fill`创建PR
2. **PR创建≠任务完成**，需满足所有以下条件才判定为完成：
   ✅ PR已创建
   ✅ 分支与main分支同步，无代码冲突
   ✅ CI全通过（lint、类型检查、单元测试、E2E测试）
   ✅ 三个AI审查者均通过
   ✅ UI改动需附带截图

### 步骤5：自动化Code Review，三Agent协同审查
每个PR由3个Agent从不同维度审查，直接在PR中留下评论，各司其职：
1. **Codex Reviewer（最靠谱）**：发现边界情况、逻辑错误、缺失的错误处理、竞态条件，误报率极低
2. **Gemini Code Assist Reviewer（免费好用）**：发现安全问题、扩展性问题，给出具体修复建议
3. **Claude Code Reviewer（过度谨慎）**：多为过度设计建议，除非标记为“critical”，否则直接跳过

### 步骤6：自动化测试，UI改动强制附截图
CI/CD管道自动执行全量测试，确保代码质量：
1. Lint代码检查 + TypeScript类型检查
2. 单元测试
3. E2E（端到端）测试
4. Playwright测试（在与生产环境一致的预览环境运行）
5. **强制规则**：PR含UI改动时，必须在描述中附带截图，否则CI直接失败（大幅缩短人工review时间）

### 步骤7：人工Review，5-10分钟快速完成
当所有自动化环节通过后，开发者收到Telegram通知，此时：
- CI全绿、三个AI审查者均批准
- UI改动有截图、边界情况已在评论中记录
- 开发者无需逐行看代码，仅需5-10分钟核查，部分PR可直接看截图合并

### 步骤8：PR合并，自动清理开发环境
1. 开发者完成人工Review后合并PR
2. 每日通过cron任务**自动清理孤立的git worktree和JSON任务记录**，保持开发环境整洁

## 五、三个让系统更“聪明”的核心机制
### 机制1：改进版Ralph Loop — 从“重复执行”到“动态学习”
传统Ralph Loop（拉取上下文→生成输出→评估结果→保存学习）的问题：**prompt静态不变**，仅优化上下文检索，未解决prompt适配问题。

#### 本系统的优化点
1. Agent失败时，Zoe会**结合业务上下文分析失败原因**，重写精准prompt而非重复原指令
   - 坏例子（静态）：`实现自定义模板功能`
   - 好例子（动态）：`停。客户要的是 X,不是 Y。这是他们在会议里的原话:'我们希望保存现有配置,而不是从头创建新的。' 重点做配置复用,不要做新建流程。`
2. **主动找活干**，无需开发者分配任务：
   - 早上：扫描Sentry错误日志，启动Agent修复bug
   - 会议后：扫描会议记录，提取客户需求并启动Agent开发
   - 晚上：扫描git log，启动Agent更新更新日志和客户文档
3. **记录成功模式**：将有效prompt结构、Agent适配规则等记录下来，奖励信号为「CI通过+AI审查通过+人工合并」，失败则触发学习循环，越用越精准。

### 机制2：Agent选择策略 — 按任务类型匹配“专家”
不同Code Agent的能力有侧重，Zoe会根据任务类型**自动选择最优Agent**，实现效率最大化，核心策略：
1. **Codex(gpt-5.3-codex) — 主力（90%任务）**
   - 擅长：后端逻辑、复杂bug修复、多文件重构、跨代码库推理
   - 特点：慢但彻底，适合核心业务开发
2. **Claude Code(claude-opus-4.5) — 速度型选手**
   - 擅长：前端开发、git操作
   - 特点：权限问题少，开发速度快（作者在Codex 5.3推出前的主力）
3. **Gemini — 设计师**
   - 擅长：UI设计，有审美能力
   - 用法：先由Gemini生成HTML/CSS设计规范，再由Claude Code实现组件

### 机制3：系统性能瓶颈 — 内存（RAM）而非Token/API速率
意外发现：系统的核心限制不是API调用成本、速率，而是**内存大小**，原因：
每个Agent运行时需要独立的：
- git worktree分支环境
- node_modules依赖包
- 代码构建/类型检查/测试进程

**资源占用**：5个Agent同时运行 = 5个TypeScript编译器 + 5个测试运行器 + 5套依赖加载到内存
**硬件限制**：16GB RAM的Mac Mini最多同时运行4-5个Agent，再多会出现内存交换，影响速度
**解决方案**：作者已购置128GB RAM的Mac Studio M4 Max（$3500），3月底到货后将验证性能提升。

## 六、快速搭建：从零到运行仅需10分钟
### 核心方法
将本文内容复制给OpenClaw，下达指令：`按照这个架构，给我的代码库实现一套 Agent 集群系统。`
OpenClaw会自动完成：
1. 读取架构设计并适配你的代码库
2. 创建自动化脚本（git/tmux/cron）
3. 设置目录结构
4. 配置cron监控任务

### 前置准备
1. OpenClaw账号
2. Codex和/或Claude Code的API访问权限
3. 一个git代码仓库
4. （可选）Obsidian，用于存储业务上下文（会议记录/客户数据等）
